2.1 IAM Fundamentals — Users / Groups / Roles / Policies
--------------------------------------------------------

Core Concept
------------

Security ဆိုတဲ့ စကားလုံးက IT ကို လုပ်နေရင် အမြဲကြားနေရတဲ့ စကားလုံး တစ်ခုပါ။ ဒါပေမယ့် Security ဆိုတာ Security Team ချည်းပဲ ကိုင်တွယ်ရမယ့် အလုပ်လို့ ထင်နေလို့မရပါဘူး။ Cloud ပေါ်မှာ လုပ်နေတဲ့ service တိုင်း Actionတိုင်းက Security Best Practice ကို လိုက်နာဖို့ လိုပါတယ်။

ကျွန်တော်တို့ ရုံးမှာ ဆိုရင် ဘယ်သူက ဘာလုပ်ခွင့်ရှိတယ်ဆိုပြီး လုပ်ပိုင်ခွင့်တွေကို သတ်မှတ်ထားလေ့ရှိပါတယ်။ ဥပမာ — Cloud Team Manager တစ်ယောက်က ဝန်ထမ်းအသစ်ဝင်လာတဲ့အခါ IT Manager ဆီကို laptop တောင်းပေးပိုင်ခွင့်ရှိပေမယ့် ရုံးကို အခုမှ ရောက်ဖူးတဲ့ ဝန်ထမ်းအသစ်မှာတော့ အဲ့လို လုပ်ပိုင်ခွင့်မရှိဘူးပေါ့။

AWS မှာလည်း ဒီအတိုင်းပါပဲ — ဘယ်သူက ဘယ် Resource ကို ဘာ Permission နဲ့ access ရမလဲဆိုတာ သတ်မှတ်ပေးဖို့ **IAM** ကို သုံးပါတယ်။

IAM OverView
---------------------------

**IAM** ဆိုတာ **Identity and Access Management** ကို အတိုကောက်ခေါ်တာပါ။ ကျွန်တော်တို့ AWS Cloud ကို သုံးနေသမျှ IAM နဲ့ ကင်းဝေးလို့မရပါဘူး။ ဒါကြောင့် ဒီ service ကို ကောင်းကောင်းနားလည်ဖို့ အရမ်းအရေးကြီးပါတယ်။ ဒါက SAA Exam အတွက်တင်မဟုတ်ပဲ လက်တွေ့လက်ငန်းခွင်မှာရော တစ်ခြား Exam တွေအတွက်ရော အရေးကြီးတာဖြစ်ပါတယ်။

**မှတ်ထားရမယ့် အချက် ၃ ချက်:**

*   IAM ဟာ **Global Service** ဖြစ်တယ်။ ဆိုလိုချင်တာက Region တစ်ခုထည်းအတွက်သက်ရောက်တာမဟုတ် ပဲ ကိုယ့်ရဲ့ AWS account ရဲ့ Service အကုန်လုံးကို သက်ရောက်တာလို့မှတ်လို့ရတယ်။

    
*   IAM ကို သုံးလို့ **ပိုက်ဆံ လုံးဝ မကုန်ပါဘူး**။ တစ်ခြား Service တော်တော်များများကတော့ Pay As You Go ပါ။ ကြိုက်သလောက်သုံး ကျသလောက်ပေးပေါ့။
    
*   Default Behaviour က **Deny**။ ဆိုလိုတာက ကိုယ် explicitly allow (မြန်မာလိုပြောရရင် တိတိကျကျ သတ်မှတ်ပေးရတာကိုပြောချင်တာ)မပေးမချင်း ဘာမှ လုပ်လို့မရဘူး။ ဥပမာ - မင်းကို EC2 Instacne တွေကို Start လုပ်ခွင့်ပဲပေးမယ်။ ဒါဆို Start Action ကလွဲပြီး ကျန်တာတွေကို သူက ဘာမှ လုပ်ခွင့်မရှိတော့ဘူး။
    

IAM ဖွဲ့စည်းပုံ — Building Blocks ၄ ခု
--------------------------------------

Atom တစ်လုံးကို Proton, Neutron, Electron တွေနဲ့ ဖွဲ့စည်းထားသလိုပဲ IAM ကိုလည်း အစိတ်အပိုင်း ၄ ခုနဲ့ ဖွဲ့စည်းထားတယ်။

### 1\. Users

User ဆိုတာ AWS Console ကို login ဝင်ပြီး သုံးမယ့် လူ (သို့) Application တစ်ခုကို ကိုယ်စားပြုတဲ့ Identity ပါ။ IAM User ဆိုလိုက်ရင် **Long-term** အတွက် ရည်ရွယ်တယ်လို့ မှတ်ထားပါ။ FaceBook Account ဖွင့်ပြီးရင် တစ်ရက် နှစ်ရက် သုံးတာမျိုးမဟုတ်ပဲ အမြဲ အကြာကြီး သုံးတဲ့ သဘောပေါ့ဗျာ။

**Credential ၂ မျိုးရှိတယ်**

`   Console Access  → Username + Password (+ MFA optional) 
    CLI / API Access → Access Key ID + Secret Access Key                     (Access Key ID ≈ username, Secret Key ≈ password)   `

**သိထားရမှာက**

*   Account တစ်ခု ဖွင့်လိုက်ပြီဆိုရင် **Root User** က အလိုအလျောက် ပါလာတယ် — Full permission ရှိတယ်၊ ဒါပေမယ့် daily work အတွက် **လုံးဝ မသုံးရဘူး AWS ကလဲ ဒါကို အကြံမပေးဘူး**
    
*   Root User ချည်းသီးသန့် လုပ်နိုင်တဲ့ action တချို့ ရှိတယ်။ account ပိတ်တာ၊ billing email ပြောင်းတာ၊ support plan ပြောင်းတာ ဒါမျိုးတွေပေါ့၊ အရမ်း powerful ဖြစ်လို့ သတိထားရသင့်တယ်။
    
*   User တစ်ယောက်မှာ Access Key **အများဆုံး ၂ ခု** ထားနိုင်တယ်
    
*   Account တစ်ခုမှာ IAM User **အများဆုံး ၅,၀၀၀** ဆောက်နိုင်တယ်
    
*   User အသစ်ဆောက်ရင် **Permission မရှိသေးဘဲ** ဆောက်လိုက်တာ AWS က အဲ့တော့ သူတို့ဆီမှာ ဘာ Permission မရှိဘူး ဘာလုပ်လုပ် Deny ဖြစ်နေမှာပဲ။ အဲ့တော့ ကိုယ်တိုင် Policy ပေးမှ ရတယ်
    
*   **Least Privilege Principle** — User တစ်ယောက်ချင်းစီကို သူ့အလုပ်နဲ့ လိုအပ်တဲ့ Permission ကိုပဲ ပေးဖို့လိုတယ်၊ ပိုပြီး ပေးတာမျိုးကို ရှောင်သင့်တယ်။ ဒါကလဲ Security Best Practices တစ်ခုပေါ့။
    

### 2\. Groups

Group ဆိုတာ User တွေကို စုပြီး ထားတဲ့ container တစ်ခုလို့မြင်ကြည့်လို့ရတယ်။  Permission ကို တစ်ယောက်ချင်းစီ လိုက်ပေးနေမယ့်အစား Group တစ်ခုလုံးကို တစ်ခါတည်း ပေးနိုင်ဖို့ လုပ်ထားတာပါ။

ဥပမာ — "Developers" Group ဆောက်ပြီး EC2 permission တွေ ပေးထားလိုက်ရင် Developer အသစ်တစ်ယောက် ဝင်လာတိုင်း Group ထဲ ထည့်လိုက်တာနဲ့ Permission အကုန် အလိုအလျောက် ရသွားတာပေါ့။

**မှတ်ထားရမယ့် Rule:**

*   Group ထဲမှာ Group ထပ်ထည့်လို့ **မရဘူး**
    
*   Group ကိုယ်တိုင် **login ဝင်လို့မရဘူး**။ User ပဲ login ဝင်နိုင်တယ်။
    
*   User တစ်ယောက်ကို Group **တစ်ခုထက်ပိုပြီး** ထည့်နိုင်တယ် (ဥပမာ Min Htet ဆိုတဲ့ User က Developers Group ထဲမှာရော Audit Group ထဲမှာပါ ရှိနိုင်တယ်)
    
*   User ကို Group ထဲ မထည့်ဘဲ ထားလို့ရတယ်။ ဒါပေမယ့် Best Practice မဟုတ်ဘူး တစ်ကယ့် Operation မှာဆိုရင်ပေါ့။
    

### 3\. Roles

ကိုယ်ပိုင် NRC လိုမျိုးက IAM User ပါ။ ဘယ်သွားသွား ကိုယ့်အိတ်ထဲမှာ အမြဲရှိတယ်။ Role ကတော့ Hotel မှာ Check-in ဝင်တဲ့အချိန် ထုတ်ပေးတဲ့ Visitor Card လိုပဲ။ အလုပ်ပြီးရင် ပြန်အပ်ရတယ်။ နောက်နေ့ ပြန်လာရင် Card အသစ် ထပ်ထုတ်ရတယ်။

`   Role = Trust Policy      (ဘယ်သူ assume လုပ်ခွင့်ရှိလဲ)       + Permission Policy  (assume ပြီးရင် ဘာလုပ်ခွင့်ရှိလဲ)   `

**Temporary Credential:**

*   Role ကို assume လုပ်တိုင်း **AWS STS (Security Token Service)** က Temporary Credential ထုတ်ပေးတယ်။ ဒါကို သေချာ မှတ်ထားသင့်တယ်။
    
*   Default: **1 နာရီ** | Maximum: **12 နာရီ** (configurable)
    
*   Long-term Access Key/Password **လုံးဝမပါဘူး**
    

**Role ကို ဘယ်အချိန် သုံးလဲ:**

ဆိုကြပါဆို့ EC2 က S3 ကို access လုပ်ချင်တယ်။ EC2 Instance Role attach လုပ်တယ်။ 

Lambda က DynamoDB ကို access ချင်တယ် Lambda Execution Role attach လုပ်တယ်။ 

Account A က Account B ထဲ ဝင်ချင်တယ် Cross-account Role assume လုပ်တယ်။

ဒီလို အခြေနေမျိုးတွေမှာ password less solution ဖြစ်ပြီး secure ဖြစ်တဲ့ AWS Assume Role ကို သုံးကြတယ်လို့ ပြောလို့ရတယ်။

**⚠️** Role ကို assume လုပ်လိုက်တာနဲ့ ကိုယ်ရဲ့ မူလရှိနေတဲ့ Permission တွေက **ပျောက်သွားတယ်** ကိုယ် assume လုပ်လိုက်တဲ့ Role ရဲ့ Permission ကိုပဲ ရတယ်။ ဒါက Resource-based Policy နဲ့ ကွာတဲ့ အချက်ပါ (Resource-based Policy မှာ မူလ Permission တွေ ဆက်ရတယ်)။ ဒါကိုတော့ နောက် Advance အခန်းမှာ ဆက်ပြီး ပြောပြပေးပါမယ်။


### 4\. Policies

User, Group, Role ၃ ခုစလုံးဟာ "ဘယ်သူလဲ" (Identity) ကိုပြောတာပါ။ "ဘာလုပ်ခွင့်ရှိလဲ" ဆိုတာကို အမှန်တကယ် ဆုံးဖြတ်ပေးတာက **Policy** ပဲဖြစ်ပါတယ်။ Policy မပါရင် User/Group/Role ဟာ ဘာမှ မလုပ်နိုင်ပါဘူး။

**Policy Structure:**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3Read",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": "arn:aws:s3:::my-bucket/*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": "203.0.113.0/24"
        }
      }
    }
  ]
}
```
ဒီ JSON ထဲမှာ ပါတဲ့ Field တွေကို တစ်ခုချင်းစီ ကြည့်ကြရအောင်:
 
- **Version** — Policy language ရဲ့ version ပါ။ အမြဲတမ်း `"2012-10-17"` လို့ ထည့်ပေးရပါတယ်။ ဒီတန်ဖိုးကနေ ပြောင်းလဲတာမျိုး မရှိဘဲ fixed ဖြစ်နေတာပါ။
- **Sid** — Statement ID ပါ။ Optional ဖြစ်ပြီး Policy ထဲမှာ Statement တွေများလာရင် ဘယ် Statement က ဘာအတွက်ဆိုတာ မှတ်မိဖို့ label ပေးတဲ့ သဘောပါ။
- **Effect** — `Allow` နဲ့ `Deny` ဆိုပြီး ၂ မျိုးသာ ရှိပါတယ်။ ခွင့်ပြုမလား ပိတ်ပင်မလားဆိုတာ ဒီနေရာမှာ ဆုံးဖြတ်တာပါ။
- **Action** — ဘယ် API call ကို လုပ်ခွင့်ပေးမလဲ (သို့) ပိတ်မလဲဆိုတာ သတ်မှတ်တဲ့နေရာပါ။ ဥပမာ `s3:GetObject` ဆိုတာ S3 ထဲက Object တွေကို ဖတ်ခွင့်ပေးတာ၊ `ec2:StartInstances` ဆိုတာ EC2 ကို Start လုပ်ခွင့်ပေးတာပါ။
- **Resource** — Action ကို ဘယ် Resource ပေါ်မှာ apply လုပ်မလဲ သတ်မှတ်တာပါ။ ARN (Amazon Resource Name) format နဲ့ ရေးရပြီး `*` ထည့်ရင် အားလုံးကို ဆိုလိုတာပါ။
- **Condition** — Optional ဖြစ်ပြီး ပိုပြီး တိကျတဲ့ Rule တွေ ထပ်ထည့်ချင်ရင် သုံးတာပါ။ ဥပမာ — သတ်မှတ်ထားတဲ့ IP ကနေပဲ ဝင်ခွင့်ပြုမယ်၊ MFA ရှိမှ allow ပြုမယ်၊ business hours ထဲမှာပဲ access ရမယ် စတဲ့ condition တွေ ထည့်သွင်းနိုင်ပါတယ်။

**Policy Type ၃ မျိုး:**

```
AWS Managed Policy      → AWS ကပဲ ရေးပြီး maintain လုပ်တယ်                            (e.g. AmazonS3ReadOnlyAccess, AdministratorAccess)                            Multiple user/role မှာ attach လုပ်နိုင်တယ်  

Customer Managed Policy → ကိုယ်တိုင် create လုပ်တယ်                            Reusable — multiple user/role မှာ attach လုပ်နိုင်တယ်                            Version control ရှိတယ်

Inline Policy           → User/Group/Role တစ်ခုချင်းစီနဲ့ တိုက်ရိုက်ချိတ်တယ်                            Reuse မလုပ်နိုင်ဘူး                            Entity ကို delete လုပ်ရင် Policy ပါ ပျောက်သွားတယ်
```

**Policy Evaluation Logic — Exam မှာ အမြဲပါတဲ့ Rule:**
`   Step 1: Default → Implicit DENY (ဘာမှ မသတ်မှတ်ထားရင် Deny)  Step 2: Explicit DENY ပါလာရင် → DENY ချက်ချင်း (ဘယ် Allow ကမှ မကျော်နိုင်)  Step 3: Explicit ALLOW ပါလာရင် → ALLOW  Step 4: ဘာမှ မပါရင် → DENY (Implicit Default)  မှတ်ထားပါ: Explicit DENY > Explicit ALLOW > Implicit DENY   `

**Conditions — Common Examples:**

`   aws:SourceIp          → ဘယ် IP ကနေ call လုပ်တာလဲ restrict  aws:RequestedRegion   → ဘယ် Region ကို call လုပ်တာလဲ restrict  aws:MultiFactorAuthPresent → MFA ရှိမှ allow  ec2:ResourceTag       → Tag အလိုက် restrict   `

Security Tools — Audit အတွက်
----------------------------

**IAM Credentials Report (Account-level)**

*   Account ထဲက User တွေအားလုံးရဲ့ credential status (password, access key, MFA) ကို spreadsheet အနေနဲ့ download ရနိုင်တယ်    

**IAM Access Advisor (User-level)**

*   User တစ်ဦးချင်းစီကို ဘယ် Service တွေ permission ပေးထားလဲ၊ နောက်ဆုံး ဘယ်အချိန်မှာ access လုပ်ဖူးခဲ့တာလဲ စတာတွေကို ကြည့်နိုင်တယ်
    
*   Least privilege ကို enforce လုပ်ဖို့အတွက် IAM Access Advisor ကို ကြည့်ပြီး permission တွေကို လိုတိုးပိုလျော့လုပ်လေ့ရှိကြတယ်။
